# 食事ログPWA 初版仕様書（v1.0）

## 1. 目的

iPhone から食事内容を簡単に記録し、Gmail 経由で Google スプレッドシートに蓄積することで、以下を可能にする。

- 食事内容の振り返り
- 外食 / 自炊の傾向把握
- 食事回数やリズムの確認

## 2. システム構成

```text
[PWA（iPhone）]
    ↓（メール送信）
[Gmail]
    ↓（GASが取得）
[Google Apps Script]
    ↓（整形・反映）
[Googleスプレッドシート]
    - dataシート（生データ）
    - summaryシート（集計）
```

## 3. 機能一覧（初版）

### PWA

- 食事ログ入力
- メール送信
- 入力内容の一時保存（localStorage）
- 直近履歴表示（最大5件）

### GAS

- Gmail から対象メール取得
- CSV 解析
- スプレッドシートへの新規追加 / 更新

### スプレッドシート

- データ蓄積
- 基本集計

## 4. データモデル

### 4.1 1レコード定義（1食）

| 項目名 | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| `id` | `string` | ○ | 一意ID |
| `datetime` | `string` | ○ | 食事日時 |
| `mealType` | `string` | ○ | 食事区分 |
| `placeType` | `string` | ○ | 食事場所 |
| `items` | `string` | ○ | 食品一覧（`|`区切り） |
| `memo` | `string` | - | 任意メモ |
| `updatedAt` | `string` | ○ | 更新日時 |
| `deleted` | `boolean` | ○ | 削除フラグ |

### 4.2 ID生成ルール

形式:

```text
yyyyMMdd-HHmmss-xxx
```

例:

```text
20260328-083015-a1b
```

- `xxx` はランダム英数字3桁
- クライアント側で生成

## 5. 入力仕様（PWA）

### 5.1 入力項目

#### ① 日時（`datetime`）

- 初期値: 現在時刻
- 手動変更可能

#### ② 食事区分（`mealType`）

選択式（必須）

- 朝食
- 昼食
- 夕食
- 間食
- 夜食
- 飲み物

#### ③ 食事場所（`placeType`）

選択式（必須）

- 自炊
- 外食
- コンビニ
- スーパー
- テイクアウト
- その他

#### ④ 食品入力（`items`）

- 複数入力欄
- 初期表示: 3行
- 最大: 10行
- 空欄は送信時に除外

保存形式:

```text
バナナ|ヨーグルト|コーヒー
```

#### ⑤ メモ（`memo`）

任意テキスト

例:

- 大盛
- 夜勤明け
- 食べすぎ

### 5.2 UI構成

```text
[日時]
[食事区分]
[場所]

[食品1]
[食品2]
[食品3]
[＋追加ボタン]

[メモ]

[送信ボタン]
[クリアボタン]

---直近履歴（5件）---
```

### 5.3 入力補助

- 前回入力の内容を `localStorage` に保持
- リロード時に復元
- 直近履歴からコピー可能（第2段階で強化）

## 6. メール送信仕様

### 6.1 件名

```text
meal-log-export
```

固定

### 6.2 本文フォーマット（CSV）

```csv
id,datetime,mealType,placeType,items,memo,updatedAt,deleted
20260328-083015-a1b,2026-03-28 08:30,朝食,自炊,"バナナ|ヨーグルト|コーヒー","いつもの朝食",2026-03-28 08:31:12,false
```

### 6.3 ルール

- ヘッダ必須
- 1メール = 複数行可（初版は1行でもOK）
- `items` は `"` で囲む
- `memo` 内のカンマ対策も `"` で対応

## 7. GAS仕様

### 7.1 処理対象メール

条件:

- 件名: `meal-log-export`
- 未処理 or ラベル指定

### 7.2 処理フロー

1. メール取得
2. 本文CSV解析
3. レコードごとに処理

### 7.3 反映ルール

#### 新規

- `id` が存在しない → 追加

#### 更新

- `id` が存在 → `updatedAt` 比較
- 新しいもののみ上書き

#### 削除

- `deleted = true` → 削除フラグ更新

## 8. スプレッドシート仕様

### 8.1 dataシート

| 列 | 内容 |
| --- | --- |
| A | `id` |
| B | `datetime` |
| C | `date` |
| D | `time` |
| E | `mealType` |
| F | `placeType` |
| G | `items` |
| H | `memo` |
| I | `updatedAt` |
| J | `deleted` |
| K | `source` |
| L | `receivedAt` |

### 8.2 自動生成列

- `date = datetime` から抽出
- `time = datetime` から抽出
- `receivedAt = GAS` 処理時刻

## 9. summaryシート仕様

### 9.1 基本集計

#### ① 日別件数

- 日ごとの食事回数

#### ② `mealType` 別件数

- 朝 / 昼 / 夜 / 間食の回数

#### ③ `placeType` 別件数

- 外食 / 自炊の割合

#### ④ 外食回数

- `placeType = 外食` の件数

### 9.2 拡張余地（初版未実装）

- 品目ランキング
- 曜日別傾向
- 食事間隔

## 10. localStorage仕様

保存データ:

```json
{
  "draft": {},
  "recent": []
}
```

内容:

- `draft`: 入力中データ
- `recent`: 直近5件

## 11. エラー・例外対応

### PWA側

- `items` がすべて空 → 送信不可
- `mealType` 未選択 → 送信不可

### GAS側

- CSV不正 → スキップ
- `id` 未定義 → スキップ

## 12. 非機能要件

### 継続性重視

- 入力3タップ以内を意識
- 項目を増やしすぎない

### 信頼性

- `localStorage` で入力保護
- スプレッドシートがデータ本体

### 拡張性

- `id` ベース更新設計
- `deleted` フラグ対応

## 13. 実装優先順位

### 必須（初回完成ライン）

- PWA入力フォーム
- メール送信
- GASでCSV解析
- dataシート反映

### 余裕があれば

- 直近履歴表示
- summaryシート作成

## 14. 今回の設計の意図（重要）

この設計はあえてこうしている:

- カロリー計算なし
- 栄養分析なし
- 自動分類なし

理由:

- 「続くこと」が最優先
